# Vault Sources Structure

Authoritative reference for `vault/sources/` organization, conventions, and file naming.

## Directory Hierarchy

```
vault/sources/
├── _index.md                    # Master catalog (auto-maintained)
├── _topics.md                   # Topic taxonomy, aliases, relationships
│
├── meetings/
│   ├── _index.md                # Type catalog
│   └── [topic]/
│       ├── _index.md            # Topic catalog
│       ├── transcript-YYYY-MM-DD.md
│       ├── summary-YYYY-MM-DD.md
│       └── notes-YYYY-MM-DD.md
│
├── emails/
│   ├── _index.md
│   └── [topic]/
│       ├── _index.md
│       └── [subject-slug]-YYYY-MM-DD.md
│
├── slack/
│   ├── _index.md
│   └── [channel-name]/
│       ├── _index.md
│       └── thread-YYYY-MM-DD.md
│
├── references/
│   ├── _index.md
│   └── [topic]/
│       ├── _index.md
│       └── [name].md
│
├── jira/                        # Jira project issues (managed by /jira-dump)
│   ├── _index.md                # Type catalog with project listing
│   └── [project-key]/           # Topic = lowercase project key
│       ├── _index.md            # Aggregate stats (not per-ticket listing)
│       └── [ISSUE-KEY].md       # One file per issue
│
├── social-media/                # Social media data exports and content
│   ├── _index.md
│   └── [platform]/              # Topic = platform name (e.g., linkedin)
│       ├── _index.md
│       └── [data files]         # Raw exports (CSV, etc.)
│
├── analysis/                    # Deep analysis reports (generated by /deep-analysis)
│   ├── _index.md
│   └── [topic]/                 # Topic = analysis subject in kebab-case
│       ├── _index.md
│       └── analysis-YYYY-MM-DD.md
│
└── notes/                       # User-authored strategic guidance
    ├── _index.md
    └── [topic]/
        ├── _index.md
        └── [name].md
```

## Organization Principles

1. **Type-first hierarchy**: Top level is the source type (`meetings/`, `emails/`, `slack/`, `references/`, `notes/`, `jira/`, `social-media/`, `analysis/`)
2. **Consistent depth**: Every type follows `type/topic/[files]` - no files at the type level
3. **Kebab-case topics**: Topic folder names use kebab-case (`game-ready-process`, `dpt-generative-tpm`)
4. **Index files everywhere**: `_index.md` at every folder level for agentic navigation
5. **Topic in path, not filename**: Filenames omit the topic since it's already in the folder path

## Index Files

| File | Level | Purpose |
|------|-------|---------|
| `_index.md` | Every folder | Catalog with file list and descriptions |
| `_topics.md` | Root only | Topic taxonomy with relationships and aliases |

### Topic Index Format

```markdown
# [topic-name]

[Brief description of what this topic contains]

## Contents

| File | Date | Description |
|------|------|-------------|
| [filename] | YYYY-MM-DD | [brief description] |
```

## Topic Naming

Topic names serve as:
- **Import destinations**: The archivist skill uses topic names to file incoming documents
- **Workstream references**: `overview.md` Sources section references topics to pull context
- **Cross-referencing**: The same topic name across types groups related materials

**Best practices:**
- Use consistent names across source types for the same subject
- Choose descriptive names that remain meaningful over time
- Prefer project/meeting names over dates or generic labels

## File Naming Conventions

| Type | Pattern | Location |
|------|---------|----------|
| Transcripts | `transcript-YYYY-MM-DD[-HHMM].md` | `meetings/[topic]/` |
| Summaries | `summary-YYYY-MM-DD[-HHMM].md` | `meetings/[topic]/` |
| Meeting Notes | `notes-YYYY-MM-DD[-HHMM].md` | `meetings/[topic]/` |
| Emails | `[subject-slug]-YYYY-MM-DD[-HHMM].md` | `emails/[topic]/` |
| Slack Threads | `thread-YYYY-MM-DD[-HHMM].md` | `slack/[channel]/` |
| References | `[descriptive-name].md` | `references/[topic]/` |
| Notes | `[name].md` | `notes/[topic]/` |
| Jira Issues | `[ISSUE-KEY].md` | `jira/[project-key]/` |
| Index Files | `_index.md` | Every folder |
| Topic Taxonomy | `_topics.md` | Root only |

### Collision Handling

When multiple files share the same date:
1. First file: `transcript-2026-01-15.md`
2. Second file: add `-HHMM` timestamp from document: `transcript-2026-01-15-1400.md`
3. Same timestamp: add sequential suffix: `transcript-2026-01-15-1400-1.md`

## Source Types

### Meetings
Raw transcripts, generated summaries, and human-authored notes from meetings. All three subtypes share the same topic folder since they relate to the same meeting series.

### Emails
Imported email threads. Filed by topic (typically the project or subject area), not by sender or date.

### Slack
Thread transcripts split from Slack channel exports. Organized by channel name rather than topic, since channel names are already meaningful identifiers.

### References
General documents: process docs, specifications, guides, external materials. The default category for content that doesn't fit other types.

### Notes
User-authored strategic guidance. Unlike other sources (imported from external systems), notes are authored in-place and represent the user's own thinking.

Typically cross-functional:
- `goals/` - Annual objectives and priorities
- `methodology/` - TPM, agile, and best practices
- `strategy/` - Strategic direction and priorities

### Jira
Jira project issue dumps managed by the `/jira-dump` skill. **Not managed by the archivist** — do not import, reclassify, or restructure jira files. Topics are lowercase project keys (e.g., `remix`, `grai`). Each topic contains one markdown file per issue plus an `_index.md` with aggregate project metadata (status breakdown, components, fix versions, labels, epics, feature requests). Indexes are auto-generated by the jira-dump script.

### Analysis
Deep analysis reports generated by the `/deep-analysis` skill. **Not managed by the archivist** — reports are generated and filed by the deep-analysis skill. Topics are analysis subjects in kebab-case (e.g., `social-media-strategy`, `hiring-pipeline`). Each topic may contain multiple dated reports as the analysis evolves. Users may move reports to workstream directives or other locations at their discretion.

### Social Media
Social media platform data exports. Topics are platform names (e.g., `linkedin`). Contains raw export data (CSV files, media) organized by platform. **Not managed by the archivist** for automated ingestion — data is manually placed from platform exports. Useful for professional presence analysis, network mapping, and content strategy.
